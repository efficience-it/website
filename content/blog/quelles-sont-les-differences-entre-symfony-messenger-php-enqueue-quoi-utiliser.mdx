Pour les connaisseurs des systèmes de messages, vous devez sûrement connaître [Symfony/Messenger](https://symfony.com/doc/current/components/messenger.html) pour son système de file d'attente. Il permet notamment d'effectuer des scripts PHP en asynchrone. Le principe est simple, lorsqu'il y a un traitement à faire, plus long qu'un controller ne peut attendre, on place dans une file d'attente de traitement la tâche à effectuer, et ensuite vous avez ce que l'on appelle des Workers qui vont dépiler ces tâches, et par exemple, faire des calculs lourds, envoyer des mails, transformer des PDF, etc.

Mais laissez-nous vous parler également de Enqueue, le système de PHP qui va plus loin. Dans cet article, nous allons donner des éléments de comparaison pour vous forger une idée.

> Ces deux composants aident les applications à envoyer et recevoir des messages vers/depuis d'autres applications ou via des files d'attente de messages.

## Rappel sur les concepts

### Comment s'installent-ils ?

Tous les deux sont simples à installer sur votre projet, il suffit de faire le composer require et de l'inscrire dans le AppKernel.php, dans la liste de Bundle, donc ici pas de différence. Voici les lignes de commande que vous pouvez utiliser pour l'installer :

```bash
composer require enqueue/enqueue-bundle enqueue/fs
composer require symfony/messenger
```

Pour Symfony Messenger, la configuration se fait dans le fichier `messenger.yaml`. Vous y déclarez vos transports et le routage des messages :

```yaml
framework:
    messenger:
        transports:
            async:
                dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
                retry_strategy:
                    max_retries: 3
                    delay: 1000
                    multiplier: 2
        routing:
            'App\Message\MonMessage': async
```

Pour Enqueue, la configuration se fait dans `enqueue.yaml` et repose sur un système de factories :

```yaml
enqueue:
    default:
        transport:
            dsn: '%env(ENQUEUE_DSN)%'
        client: ~
```

### Architecture des messages

La façon dont on crée et dispatche les messages diffère entre les deux outils. Avec Symfony Messenger, un message est un simple objet PHP (un POPO) accompagné d'un handler :

```php
namespace App\Message;

class GenererRapportMessage
{
    public function __construct(
        private int $rapportId,
        private string $format = 'pdf',
    ) {}

    public function getRapportId(): int
    {
        return $this->rapportId;
    }

    public function getFormat(): string
    {
        return $this->format;
    }
}
```

```php
namespace App\MessageHandler;

use App\Message\GenererRapportMessage;
use Symfony\Component\Messenger\Attribute\AsMessageHandler;

#[AsMessageHandler]
class GenererRapportHandler
{
    public function __invoke(GenererRapportMessage $message): void
    {
        // traitement du rapport
    }
}
```

Le dispatch se fait via le `MessageBusInterface` :

```php
$this->bus->dispatch(new GenererRapportMessage(42, 'xlsx'));
```

Avec Enqueue, l'approche est différente. On travaille directement avec des objets de type `Message`, `Queue` et `Producer` issus du protocole de messaging :

```php
use Enqueue\Client\ProducerInterface;

$producer->sendEvent('generer_rapport', json_encode([
    'rapport_id' => 42,
    'format' => 'xlsx',
]));
```

L'avantage de Messenger est sa simplicité et son intégration native avec Symfony. L'avantage d'Enqueue est sa proximité avec les protocoles de messaging standards, ce qui facilite l'interopérabilité avec des systèmes non-PHP.

### Les Transports

Globalement, les deux systèmes ont les mêmes transports disponibles, alors il n'est pas forcément utile de tous les lister, mais vous avez AMQP, Kafka, MongoDB, Redis, SNS, SQS, Doctrine...

Cependant, Enqueue se distingue par un plus grand nombre de transports natifs. Il supporte notamment Google PubSub, Stomp, Amazon Kinesis, Wamp et le filesystem, là où Messenger se concentre sur les transports les plus courants. Il est aussi important de noter que Messenger permet d'écrire facilement un transport custom en implémentant l'interface `TransportInterface`, ce qui offre une grande flexibilité.

Un point intéressant concerne le transport Doctrine. Messenger utilise par défaut une table `messenger_messages` avec un système de polling, tandis qu'Enqueue utilise le protocole DBAL de manière similaire mais avec une gestion différente du verrouillage. En production, pour les deux outils, il est recommandé de privilégier un broker de messages dédié comme RabbitMQ ou Redis plutôt que la base de données relationnelle.

### Surveillance

PHP Enqueue fournit un outil de surveillance des files d'attente de messages, alors que Messenger non. Avec Enqueue, vous pouvez contrôler combien de messages ont été envoyés, combien ont été traités avec succès ou ont échoué. Combien de Workers travaillent, leur temps de disponibilité, les statistiques des messages traités, l'utilisation de la mémoire et la charge du système. L'outil peut être intégré à pratiquement n'importe quelle plateforme d'analyse et de surveillance.

Il existe plusieurs intégrations : Datadog, Grafana... donc si vous souhaitez l'essayer ou même l'intégrer, vous trouverez la page d'installation [ici](https://php-enqueue.github.io/monitoring/#installation).

Du côté de Messenger, la surveillance passe par d'autres mécanismes. Depuis Symfony 5.2, le composant intègre des événements (`WorkerMessageReceivedEvent`, `WorkerMessageHandledEvent`, `WorkerMessageFailedEvent`) que vous pouvez écouter pour alimenter vos propres métriques. Il est également possible de combiner Messenger avec le Profiler de Symfony en développement, et avec des outils comme Prometheus ou StatsD en production via des listeners personnalisés.

### Gestion des erreurs et retry

Les deux outils gèrent les messages en échec, mais de manière différente. Messenger dispose d'une stratégie de retry configurable directement dans le YAML, avec un nombre maximum de tentatives, un délai entre chaque tentative et un multiplicateur. Les messages qui échouent définitivement sont envoyés vers un transport dédié appelé "failure transport" :

```yaml
framework:
    messenger:
        failure_transport: failed
        transports:
            failed: 'doctrine://default?queue_name=failed'
```

Vous pouvez ensuite inspecter et rejouer les messages échoués via la console :

```bash
php bin/console messenger:failed:show
php bin/console messenger:failed:retry
```

Enqueue propose un mécanisme similaire via des extensions (DelayRedeliveredMessageExtension) mais nécessite davantage de configuration manuelle. La gestion des dead-letter queues est cependant plus flexible car elle s'appuie directement sur les fonctionnalités natives du broker sous-jacent.

## Les limites des files d'attente

Au-delà de son aspect pratique, il existe des bordures à ne pas dépasser dans son utilisation. Car avec de gros volumes de messages à traiter, vous pouvez trouver quelques contraintes.

Par expérience, nous nous sommes trouvés à traiter quelques centaines de milliers de messages par jour dans un projet où les messages étaient stockés dans Doctrine (via une table `messenger_messages`), et comme le système ne dépilait pas assez, nous nous sommes dit : augmentons le nombre de workers !

Grave erreur, car la concurrence des accès en base de données est très compliquée pour **Redis** comme pour **Doctrine/MySQL**.

Donc comment augmenter le dépilage sans augmenter le nombre de Workers ? Nous avons tenté d'insérer des index sur les `queue_name` et `available_at` de la table `messenger_messages`, car en décortiquant la requête SQL, nous nous sommes aperçus qu'il y avait un ORDER BY sur `available_at`.

Bonne surprise, cela a fonctionné : la pile diminuait. Sauf qu'effet de bord, nous avons remarqué que nos workers s'arrêtaient régulièrement bien avant notre `--time-limit` configuré sur la commande. (Nous lancions des workers toutes les heures avec un time-limit de 3600 secondes, ce qui permettait de s'assurer qu'ils étaient killés tôt ou tard, tout en sachant qu'il y en avait forcément qui seraient relancés).

En fait, en lançant le worker en verbose et à la main, nous avions une erreur : `Deadlock found when trying to get lock; try restarting transaction`. Le MySQL que nous avions (5.7) n'était pas prévu pour avoir des INSERT/UPDATE/DELETE avec des index comme ceux que nous avions, il fallait passer en 8.0.

Le problème était que nous dépilions pourtant bien, mais juste cette erreur ralentissait le dépilage. Ce que nous avons fait est donc très simple :

Nous avons lancé dans notre crontab des workers toutes les minutes avec un time-limit de 60 secondes, ainsi nous étions sûrs qu'ils seraient très souvent relancés malgré cette erreur, et que nous dépilerions rapidement nos listes de messages à traiter.

Pour aller plus loin, nous serions passés par [RabbitMQ](https://www.rabbitmq.com/) qui stocke la donnée en NoSQL sous son propre format, et qui permet d'aller plus vite dans le système de dépilage puisque c'est un outil créé justement pour subir des INSERT/UPDATE/DELETE de manière concurrente sans problème.

## Migrer de PHP Enqueue vers Symfony Messenger

Si vous utilisez actuellement Enqueue et que vous envisagez de migrer vers Messenger, voici quelques conseils pratiques. La migration est facilitée par le fait que les deux outils partagent les mêmes concepts fondamentaux (messages, handlers, transports).

La première étape consiste à réécrire vos processeurs Enqueue en handlers Messenger. Chaque `Processor` Enqueue devient un handler avec l'attribut `#[AsMessageHandler]`. Ensuite, remplacez les appels au `ProducerInterface` d'Enqueue par le `MessageBusInterface` de Messenger.

Il est recommandé de procéder transport par transport plutôt que de tout migrer d'un coup. Vous pouvez faire cohabiter les deux systèmes temporairement en gardant Enqueue pour certaines queues et Messenger pour d'autres, le temps de valider que chaque migration fonctionne correctement en production.

Un point d'attention important : si vous utilisez les fonctionnalités de monitoring d'Enqueue, prévoyez de mettre en place une solution de remplacement (listeners Symfony + outil de métriques) avant de désactiver Enqueue.
