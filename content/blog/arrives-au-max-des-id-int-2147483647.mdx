## Erreur MySQL « Out of range value » : éviter la saturation des IDs INT

Tôt ou tard dans la carrière d'un développeur ou d'un DevOps, on atteint le maximum d'un identifiant INT dans MySQL. C'est alors la panique : plus d'insertions possibles, applications bloquées, et logs en erreur.

Chez Efficience IT, cette situation survient régulièrement sur des applications en production à fort trafic. Anticiper ce problème s'avère essentiel pour préserver la continuité de service et éviter une panne critique.

Bien que ce scénario semble rare, dès qu'un projet produit un grand volume de données (emails, logs, événements, historiques...), la limite des **2 147 483 647** lignes est vite atteinte.

Ce chiffre correspond à la valeur maximale d'un champ `INT SIGNED`. Avec `UNSIGNED`, la capacité double, mais plafonne à environ **4,2 milliards d'entrées**. Au-delà, la migration vers **BIGINT** s'impose, capable de gérer jusqu'à `2^64 - 1` valeurs.

## Précautions à prendre après une première erreur

Quand on rencontre MySQL de cette manière, on apprend à respecter sa mécanique interne.

- **Tuer un processus MySQL** ne libère pas immédiatement les ressources. MySQL effectue une restauration interne, très coûteuse en CPU et en I/O.
- Lors de tests mal planifiés, jusqu'à **48 heures de récupération complète** ont été observées, même sur une infrastructure correctement dimensionnée.
- Les grands `SELECT` ou `INSERT` peuvent bloquer les insertions concurrentes selon le niveau d'isolation de transaction.

Même avec l'option `SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;`, les performances chutent. D'où l'importance de planifier une migration maîtrisée.

## Comment migrer un champ INT vers un BIGINT

Passer de `2^32 - 1` à `2^64 - 1` peut se faire de plusieurs manières, selon :

- la taille de votre table
- le volume de données
- la tolérance à l'interruption

Dans tous les cas, commencez par exécuter ces deux commandes :

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SET FOREIGN_KEY_CHECKS = 0;
```

La première ne verrouille pas les lignes pendant la copie. La seconde réduit la charge en ignorant la vérification de clés externes.

Trois possibilités s'offrent à vous suivant le temps disponible et la taille de votre table. Dans tous les cas, **testez impérativement sur une base de tests avec une machine de puissance égale à votre production**.

## Solution 1 : Déplacer les ID (table inférieure à 1 Go)

Méthode simple pour gagner du temps sans refonte majeure : déplacer l'auto-incrément de la base de données. On configure l'INT pour qu'il commence à 0 et termine autour de 2^33, doublant ainsi la capacité.

```sql
ALTER TABLE table
CHANGE `id` `id` INT(11) UNSIGNED NOT NULL AUTO_INCREMENT,
ALGORITHM=COPY, LOCK=SHARED;
```

Vous doublez la capacité de votre ID (environ 4 294 967 294 lignes).

**Durée** : 1 à 5 minutes selon la taille de la table.

Idéal pour les petites bases avec peu de dépendances.

## Solution 2 : Passer en BIGINT (table inférieure à 1 Go)

Modifiez la colonne directement en BIGINT :

```sql
ALTER TABLE table
CHANGE `id` `id` BIGINT(12) UNSIGNED NOT NULL AUTO_INCREMENT,
ALGORITHM=COPY, LOCK=SHARED;
```

L'avantage est la simplicité de la commande. L'inconvénient est un temps d'exécution potentiellement long pour des tables moyennes (plusieurs heures possibles).

Testez impérativement sur un environnement de recette avant de passer en production.

## Solution 3 : Échanger les colonnes (table de quelques Go)

Méthode progressive et sûre, idéale pour les tables de taille intermédiaire.

### Ajouter une nouvelle colonne

```sql
ALTER TABLE table ADD `new_id` BIGINT(12) UNSIGNED NOT NULL;
```

### Copier les données par paquets

```sql
UPDATE table SET new_id = id WHERE new_id = 0 LIMIT 10000;
```

### Renommer les colonnes et les clés

```sql
ALTER TABLE table DROP PRIMARY KEY;
ALTER TABLE table CHANGE id old_id INT;
ALTER TABLE table CHANGE new_id id BIGINT UNSIGNED AUTO_INCREMENT, ADD PRIMARY KEY(id);
```

### Supprimer la colonne old_id

Une fois la migration validée, supprimez l'ancienne colonne devenue inutile.

Ce processus, automatisé avec un script batch, minimise la charge et garantit la cohérence des clés étrangères.

## Solution 4 : Échanger les tables (table de 10 Go et plus)

Pour les très gros volumes, créer une nouvelle table miroir s'avère plus fiable.

```sql
CREATE TABLE table_v2 LIKE table;
ALTER TABLE table_v2 CHANGE id id BIGINT(12) UNSIGNED NOT NULL AUTO_INCREMENT;
```

Définissez l'incrément de départ :

```sql
SELECT AUTO_INCREMENT FROM information_schema.TABLES
WHERE TABLE_NAME='table' AND TABLE_SCHEMA='database';
ALTER TABLE table_v2 AUTO_INCREMENT = 2147483648;
```

Puis insérez les données par paquets :

```sql
INSERT INTO table_v2 SELECT * FROM table LIMIT 10000;
```

Renommez ensuite :

```sql
RENAME TABLE table TO table_old, table_v2 TO table;
```

Cette solution offre une migration complète avec une sécurité maximale. Le processus est plus long et nécessite un suivi rigoureux, mais c'est la méthode la plus fiable pour les bases volumineuses.

## Ce que recommande Efficience IT

Chaque scénario dépend du volume de données, du temps de maintenance disponible et de la tolérance au risque.

Notre approche :

- **Audit préalable** de la base MySQL et des dépendances applicatives
- **Simulation sur environnement miroir** pour valider la procédure
- **Migration progressive** avec rollback planifié à chaque étape
- **Optimisation post-migration** : index, performances, monitoring
