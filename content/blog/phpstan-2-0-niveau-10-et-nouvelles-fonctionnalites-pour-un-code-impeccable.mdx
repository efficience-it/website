L'analyse statique PHP franchit un nouveau cap avec PHPStan 2.0. Cette version majeure apporte un niveau d'analyse plus strict (niveau 10), une meilleure gestion des types génériques et une réduction de la consommation mémoire. Si vous utilisez déjà PHPStan au quotidien, cette mise à jour mérite toute votre attention. Si vous ne l'avez pas encore adopté, c'est le moment idéal pour franchir le pas.

## Le niveau 10 : une nouvelle dimension

Le niveau 9 offrait déjà une analyse rigoureuse détectant les incohérences de typage, les propriétés non initialisées et les conditions inutiles. Le niveau 10 va plus loin avec des vérifications détaillées : détection des dépendances inutiles et des appels dynamiques non sécurisés, parfait pour les projets critiques.

### Le problème du type mixed implicite

Bien que le niveau 6 demande d'ajouter des types manquants, certaines situations permettaient de contourner cette exigence. Seul le niveau 10 détecte les types implicitement « mixed » avec précision.

Prenons un exemple concret. Avec un paramètre non typé, PHPStan au niveau 9 ne signale aucune erreur :

```php
function processData($data)
{
    return $data->getName();
}
```

Au niveau 10, PHPStan signale que `$data` a un type implicitement `mixed`. Il est impossible d'appeler `getName()` sur un type `mixed` sans risque. La correction attendue est simple :

```php
function processData(User $data): string
{
    return $data->getName();
}
```

### Détection des opérations sur mixed

Le niveau 10 détecte aussi les opérations arithmétiques sur des types `mixed`. Ce code passe au niveau 9 sans erreur :

```php
function calculateTotal(array $items): int
{
    $total = 0;
    foreach ($items as $item) {
        $total += $item['price'] * $item['quantity'];
    }
    return $total;
}
```

Au niveau 10, PHPStan signale que `$item` est de type `mixed`, rendant l'accès aux clés `price` et `quantity` non sécurisé. La solution consiste à typer le paramètre avec un PHPDoc :

```php
/**
 * @param array<array{price: int, quantity: int}> $items
 */
function calculateTotal(array $items): int
{
    $total = 0;
    foreach ($items as $item) {
        $total += $item['price'] * $item['quantity'];
    }
    return $total;
}
```

Ce type de vérification élimine une catégorie entière de bugs qui ne se manifestent qu'en production.

## Des listes plus précises pour un code plus sûr

PHPStan 2.0 introduit le concept de « liste » pour représenter des tableaux aux clés entières consécutives commençant à 0. L'annotation `@param list<int>` indique clairement qu'un tableau d'entiers avec des clés numériques consécutives est attendu.

### Différence entre array et list

En PHP, `array_values()` retourne toujours une liste, mais un simple `array<int, string>` ne garantit pas que les clés commencent à 0. Avec le type `list`, PHPStan peut détecter les incohérences :

```php
/**
 * @param list<string> $names
 */
function displayNames(array $names): void
{
    foreach ($names as $index => $name) {
        echo "$index: $name\n";
    }
}

$users = [3 => 'Alice', 7 => 'Bob'];
displayNames($users);
```

PHPStan signale que `$users` n'est pas une `list<string>` car les clés ne sont pas consécutives à partir de 0. Cette distinction réduit les bugs liés aux manipulations de tableaux, fréquents en PHP.

## Moins de mémoire, plus de vitesse

PHPStan 2.0 est devenu bien plus performant. Grâce à des améliorations sur les arbres abstraits (AST) et l'élimination des fuites mémoire, les temps d'analyse ont considérablement diminué.

L'analyse du code de PrestaShop 8.0, qui prenait 9 minutes, ne prend plus que 3 minutes, tout en consommant 50 à 70 % de mémoire en moins.

Ces gains proviennent de plusieurs optimisations internes. Les nœuds AST utilisent désormais moins d'objets intermédiaires. Le ramasse-miettes de PHP est mieux exploité grâce à une réduction des références circulaires. Sur des projets de grande taille comportant des centaines de milliers de lignes, la différence est significative et rend l'intégration dans une CI bien plus fluide.

## Cohérence des annotations @var

Lorsqu'une annotation `@var` est utilisée, PHPStan 2.0 vérifie si l'information correspond au type réel de la variable. En cas de divergence, une erreur potentielle est signalée.

Voici un cas typique où l'annotation `@var` est incorrecte :

```php
/** @var string $count */
$count = $repository->count([]);
```

Ici, `count()` retourne un `int`, mais l'annotation déclare un `string`. PHPStan 2.0 détecte cette incohérence et remonte une erreur. Auparavant, l'annotation `@var` était considérée comme une vérité absolue, ce qui pouvait masquer des bugs.

Ce changement encourage à supprimer les annotations `@var` inutiles et à se fier aux types déduits automatiquement par PHPStan.

## Fonctions pures

Les fonctions pures offrent testabilité, réutilisabilité et parallélisme. L'annotation `@phpstan-pure` assure que les fonctions respectent ces principes, ne modifiant pas l'état du programme.

### Vérification stricte des effets de bord

Lorsqu'une fonction est annotée `@phpstan-pure`, PHPStan vérifie qu'elle ne produit aucun effet de bord. Par exemple :

```php
/** @phpstan-pure */
function formatPrice(int $cents): string
{
    return number_format($cents / 100, 2, ',', ' ') . ' €';
}
```

Cette fonction est pure : elle ne modifie aucune variable extérieure et retourne toujours le même résultat pour les mêmes paramètres. En revanche, le code suivant provoque une erreur :

```php
/** @phpstan-pure */
function formatAndLog(int $cents, Logger $logger): string
{
    $logger->info('Formatting price');
    return number_format($cents / 100, 2, ',', ' ') . ' €';
}
```

PHPStan signale que `$logger->info()` est un effet de bord incompatible avec l'annotation `@phpstan-pure`. Cette vérification aide à maintenir une architecture propre et prévisible.

## Un cache plus léger

PHPStan 2.0 a revu son système de cache : analyses plus rapides, consommation mémoire réduite et gain d'espace disque en supprimant les données inutiles.

Le format de cache a été entièrement refondu. Les fichiers de cache sont désormais plus compacts et plus rapides à charger. Sur les projets volumineux, le répertoire de cache peut passer de plusieurs centaines de mégaoctets à quelques dizaines. Ce gain se traduit aussi par des pipelines CI plus rapides, car le cache se restaure et se génère plus vite.

## Migrer de PHPStan 1.x à 2.0

La migration vers PHPStan 2.0 nécessite quelques ajustements. Voici les points principaux à prendre en compte.

### Mise à jour des dépendances

La commande de mise à jour est classique :

```bash
composer require --dev phpstan/phpstan:^2.0
```

Les extensions tierces (phpstan-symfony, phpstan-doctrine, phpstan-phpunit) doivent aussi être mises à jour vers leurs versions compatibles 2.0. Vérifiez la compatibilité de chaque extension avant la migration.

### Changements de comportement

Certaines règles ont changé de niveau. Des erreurs qui n'apparaissaient qu'au niveau 8 peuvent désormais apparaître au niveau 7. Il est conseillé de relancer l'analyse complète après la mise à jour et de mettre à jour le fichier baseline si vous en utilisez un :

```bash
phpstan analyse --generate-baseline
```

### Suppression des fonctionnalités dépréciées

PHPStan 2.0 supprime les options de configuration dépréciées depuis la version 1.x. Si votre fichier `phpstan.neon` utilise des clés obsolètes, PHPStan affiche un message d'erreur clair indiquant la nouvelle syntaxe à adopter.

## Conclusion

Porté par une communauté passionnée et un écosystème d'extensions florissant, PHPStan s'impose comme la référence de l'analyse statique PHP. L'ajout du support PHP 8.4 marque une nouvelle étape de son évolution. Le niveau 10, les vérifications de cohérence `@var`, les fonctions pures et les gains de performance font de cette version 2.0 une mise à jour incontournable pour tout projet PHP soucieux de la qualité de son code.
