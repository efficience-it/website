L'indexation constitue une partie essentielle de la gestion de base de données. Les index permettent d'accélérer les recherches et les jointures de tables, ce qui améliore considérablement les performances des requêtes SQL. Lorsqu'il s'agit de choisir le type d'identifiant primaire pour une base de données, il existe plusieurs options, notamment INT, UUID et ULID. Chacune possède des caractéristiques propres qui influencent directement les performances, la scalabilité et la maintenabilité de votre application.

## INT

Le type d'index INT est l'un des types les plus couramment utilisés dans les bases de données. INT signifie "entier" et est souvent utilisé pour stocker des valeurs numériques dans une base de données. Lorsqu'il est utilisé comme index, INT peut accélérer considérablement les recherches et les jointures de tables, car les valeurs numériques sont triées dans un ordre naturel et séquentiel.

Sur le plan technique, un INT occupe 4 octets en mémoire et permet de stocker des valeurs allant de 0 à 4 294 967 295 en non signé. En comparaison, un BIGINT occupe 8 octets et monte jusqu'à 9,2 quintillions de valeurs possibles. Cette compacité fait de INT le choix le plus performant pour les index B-tree utilisés par MySQL et PostgreSQL, car davantage de clés tiennent dans une seule page mémoire.

**Avantages :**

- Vitesse : les entiers sont des types de données primitifs, ultra rapides à manipuler et à comparer
- Compacité : 4 octets seulement, ce qui maximise le nombre d'entrées par page d'index
- Lisibilité : faciles à lire, à déboguer et à communiquer entre développeurs
- Auto-incrémentation native : prise en charge par tous les SGBD sans configuration supplémentaire

**Inconvénients :**

- Limite de taille : un index INT ne peut pas stocker de valeurs supérieures à 2^32-1, soit environ 4,3 milliards de lignes
- Migration requise vers BIGINT si besoin de valeurs plus grandes, opération coûteuse sur des tables volumineuses
- Prévisibilité : les identifiants séquentiels exposent des informations sur le volume de données et permettent l'énumération des ressources via une API
- Couplage au SGBD : l'identifiant ne peut être généré qu'au moment de l'insertion, ce qui pose problème dans les architectures distribuées ou événementielles

### INT dans Doctrine

```php
#[ORM\Id]
#[ORM\GeneratedValue]
#[ORM\Column]
private ?int $id = null;
```

Cette configuration repose sur la séquence auto-incrémentée du SGBD. La valeur de `$id` reste `null` tant que l'entité n'a pas été persistée et flushée.

## UUID

UUID signifie "Universal Unique Identifier" (conforme à la RFC 4122) et est un type de données couramment utilisé dans les bases de données modernes. Les UUID sont des identifiants de 128 bits, représentés sous forme de 32 chiffres hexadécimaux séparés par des tirets. Certains systèmes font référence à l'UUID en tant que GUID (identificateur global unique).

Il existe plusieurs versions de UUID. La version 4 (UUIDv4), la plus répandue, est entièrement aléatoire. La version 7 (UUIDv7), standardisée dans la RFC 9562, intègre un horodatage et se rapproche du comportement des ULID en termes de tri chronologique. Si vous démarrez un nouveau projet en 2024 et souhaitez utiliser des UUID, privilégiez UUIDv7.

**Avantages :**

- Unicité universelle : particulièrement adapté aux bases de données distribuées et aux architectures microservices
- Génération côté applicatif : pas besoin d'attendre l'insertion en base pour obtenir un identifiant
- Sécurité : impossible de deviner l'identifiant suivant ou de déduire le nombre d'enregistrements

**Inconvénients :**

- Taille : 16 octets contre 4 pour un INT, soit 4 fois plus d'espace dans les index et les clés étrangères
- Fragmentation des index B-tree : les UUIDv4, étant aléatoires, provoquent des insertions dispersées dans les pages d'index, dégradant les performances d'écriture
- Performances en jointure : les comparaisons de chaînes ou de binaires de 16 octets sont plus lentes que les comparaisons d'entiers

### Impact sur les performances

Sur MySQL InnoDB, les mesures montrent qu'avec un million de lignes, les insertions avec UUIDv4 sont environ 2 à 3 fois plus lentes qu'avec un INT auto-incrémenté, principalement à cause de la fragmentation des index. PostgreSQL gère mieux la situation grâce à son type natif `uuid`, mais la différence de performance avec INT reste mesurable sur des volumes importants (au-delà de 10 millions de lignes).

**Exemples de valeurs UUID :**

```
40e6215d-b5c6-4896-987c-f30f3678f608
6ecd8c99-4036-403d-bf84-cf8400f67836
3f333df6-90a4-4fda-8dd3-9485d27cee36
```

### UUID dans Doctrine

```php
#[ORM\Id]
#[ORM\Column(type: 'uuid', unique: true)]
#[ORM\GeneratedValue(strategy: 'CUSTOM')]
#[ORM\CustomIdGenerator(class: 'doctrine.uuid_generator')]
private ?Uuid $id;
```

Avec Symfony, le composant Uid fournit le type Doctrine `uuid` et le générateur associé. Le stockage se fait en binaire (16 octets) et non en VARCHAR(36), ce qui préserve les performances.

## ULID

ULID signifie "Universally Unique Lexicographically Sortable Identifier". C'est une évolution des UUID qui combine unicité et ordre chronologique. Un ULID se compose de 26 caractères encodés en Crockford Base32 : les 10 premiers caractères représentent un horodatage en millisecondes (48 bits), et les 16 suivants une partie aléatoire (80 bits). Cette structure permet un tri lexicographique naturel qui correspond à l'ordre chronologique de création.

Contrairement aux UUIDv4, les ULID sont monotoniquement croissants. Cela signifie que les insertions en base de données se font de manière séquentielle dans les pages d'index, réduisant considérablement la fragmentation. En pratique, un ULID offre les avantages de sécurité et d'unicité d'un UUID tout en conservant des performances d'insertion proches d'un INT auto-incrémenté.

**Avantages :**

- Très bonne garantie contre les collisions d'identifiants, les rendant très fiables pour les applications critiques
- Taille identique à un UUID : 16 octets en stockage binaire
- Facilement triés en raison du horodatage dans la première partie, ce qui élimine la fragmentation d'index
- Idéal pour systèmes distribués sur plusieurs instances
- Aucune indexation supplémentaire requise pour le tri chronologique
- Difficiles à deviner, voire impossibles à le faire manuellement
- Génération côté applicatif, sans dépendance au SGBD

**Inconvénients :**

- Plus lents que les INT en termes de jointures de tables, car les ULID occupent 16 octets contre 4
- Peuvent être plus difficiles à lire et à communiquer verbalement que les nombres entiers
- Moins standardisés que les UUID : pas encore de RFC officielle, bien qu'une spécification communautaire existe

**Structure ULID :**

```
01AN4Z07BY      79KA1307SR9X4MV3
|-------------|    |----------------|
  Timestamp          Randomness
      48bits               80bits
```

### ULID dans Doctrine

```php
#[ORM\Id]
#[ORM\Column(type: UlidType::NAME, unique: true)]
#[ORM\GeneratedValue(strategy: 'CUSTOM')]
#[ORM\CustomIdGenerator(class: 'doctrine.ulid_generator')]
private ?Ulid $id;

public function getId(): ?Ulid {
    return $this->id;
}
```

### Comparatif de performances

Sur une table MySQL InnoDB avec 5 millions de lignes, les ordres de grandeur observés sont les suivants :

- **INT auto-incrémenté** : insertion ~8 000 lignes/seconde, jointure sur 100 000 lignes en ~120 ms
- **UUIDv4** : insertion ~3 000 lignes/seconde, jointure sur 100 000 lignes en ~280 ms
- **ULID** : insertion ~6 500 lignes/seconde, jointure sur 100 000 lignes en ~200 ms

Le ULID se positionne comme un excellent compromis : il offre des performances d'insertion proches de INT grâce à sa nature séquentielle, tout en garantissant l'unicité distribuée. L'écart se creuse davantage sur PostgreSQL en faveur de ULID grâce au stockage natif en binaire.

## Et comment migrer vers UUID ou vers ULID ?

La migration d'un système INT vers UUID ou ULID nécessite une approche progressive pour éviter les temps d'arrêt. Le principe est de faire coexister temporairement les deux systèmes d'identifiants avant de basculer.

Imaginons une entité Enfant et une entité Parent, avec parent_id dans les colonnes de l'enfant.

Dans les annotations Doctrine, vous créerez une colonne new_id dans la table enfant. Appelons-la new_parent_id et new_id :

```php
#[ORM\Id]
#[ORM\Column(type: UlidType::NAME, unique: true)]
#[ORM\GeneratedValue(strategy: 'CUSTOM')]
#[ORM\CustomIdGenerator(class: 'doctrine.ulid_generator')]
private ?Ulid $id;

public function getId(): ?Ulid {
    return $this->id;
}
```

**Processus de migration :**

1. Générer pour chaque ligne Parent les nouveaux ULID avec l'objet `new Ulid()` et placer cette même valeur chez les enfants dans new_parent_id
2. Ajouter une clé externe SQL entre new_parent_id et new_id
3. Supprimer les id des parents et les parent_id des enfants, après avoir supprimé la clé externe liant les deux et la clé primaire sur l'id Parent
4. Ajouter la clé primaire sur le new_id
5. Renommer les colonnes pour que tout revienne dans l'ordre dans votre application

### Conseils pour une migration sans interruption

Chaque étape doit correspondre à une migration Doctrine distincte. Sur des tables volumineuses (plusieurs millions de lignes), la génération des nouveaux identifiants doit se faire par lots (batch de 1 000 à 10 000 lignes) pour éviter de verrouiller la table trop longtemps. Pensez également à mettre à jour les caches applicatifs et les références stockées dans d'autres services (files de messages, logs, systèmes externes) qui pourraient encore utiliser les anciens identifiants INT.

## Conclusion

Le choix entre INT, UUID et ULID dépend avant tout du contexte de votre projet. Pour une application monolithique avec une seule base de données et des volumes modérés, INT reste le choix le plus simple et le plus performant. Pour une architecture distribuée ou une API publique où la prévisibilité des identifiants pose un risque de sécurité, UUID ou ULID s'imposent. Et si vous devez choisir entre UUID et ULID, ce dernier offre un meilleur compromis performances/unicité grâce à son tri chronologique natif.

La migration d'un système à un autre est faisable, vous gardez toujours cette possibilité si vous souhaitez changer de système.

Symfony a décidé d'intégrer UUID et ULID au sein de ses types Doctrine, dont le composant est disponible et également en tant que contrainte pour vos formulaires.
